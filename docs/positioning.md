# ForkFirst Positioning

This is the source of truth for how ForkFirst should explain itself on the landing page, README, demos, and public launch copy.

## Core Belief

People waste a lot of time and AI tokens building from a blank page. This is especially painful for inexperienced AI-builder users: they may not know what already exists, what to search for on GitHub, or how to give an AI builder a strong foundation.

ForkFirst exists because many app ideas already have something close on GitHub: a working product, useful code structure, data model, UI pattern, or repo that can be cloned, customized, and adapted. The AI builder performs better when it starts from real code and clear direction instead of guessing from scratch.

## Product Story

A user gets an app idea and comes to ForkFirst before asking an AI builder to code.

They chat with ForkFirst like they would with ChatGPT. ForkFirst understands the project, searches GitHub, and helps answer: what already exists, what is worth using as a foundation, what should only be studied, and what should be avoided.

Then ForkFirst creates the prompt and handoff files for the user's chosen AI builder. The handoff tells the builder which repo to clone or open, what working parts to keep, what to replace, and what to build first.

The result should feel like: the user can leave with a clear build packet and come back to a working prototype or product that started from something real, then got customized for their dream and needs.

## Primary Audience

Founder-first, builder-credible.

ForkFirst should first reassure people who have app ideas and use AI tools but are not deep GitHub or open-source experts. The product should also retain credibility for technical builders using Claude Code, Codex, Cursor, Replit, Lovable, v0, Gemini CLI, Antigravity, or similar tools.

## Plain-English Promise

Do not make your AI builder start from zero.

Talk through your app idea. ForkFirst finds real GitHub projects that can become your foundation, then creates the repo, prompt, and handoff files your AI builder needs to clone, customize, and build your version faster.

## Outcomes To Emphasize

- Save tokens.
- Avoid blank-page builds.
- Start from code that already works.
- Give the AI builder direction.
- Help non-technical users discover what is already out there.
- Turn a rough idea into a repo-backed build packet.
- Customize and tweak a working foundation for the user's needs.

## Language Guidelines

- Prefer: "real GitHub projects", "working foundation", "starting point", "repo, prompt, and handoff files", "AI builder".
- Use sparingly: "open source", "Markdown", "repo-root", "license signals".
- Avoid hero-level jargon: "Markdown-friendly", "repo evidence", "foundation decision", "open-source starting point".
- Do not claim legal/license safety. Say "inspect", "confirm", or "check" license and setup before reuse.
- Do not imply ForkFirst directly clones repos unless that behavior exists in the app. Say it gives the AI builder what it needs to clone or open the chosen foundation.
